# 待完成功能清单

本文把当前待建设能力拆分为可交付的工作项和验收标准。范围限于投研分析任务的配置、调度和数据访问；不涉及真实交易下单、券商账户或资金操作。

## 建议实施顺序

1. 完成 LLM 数据库配置，建立配置版本和任务快照。
2. 在保留 PostgreSQL 任务状态的前提下，引入 Redis 队列和独立 worker，实现多节点消费。
3. 增加分析任务的暂停、恢复和受控重试。
4. 对数据源做基准验证，固化各工具的优先级。
5. 基于实际调用指标处理免费数据源限流、缓存和替换决策。

第 1 项应先于第 2 项完成：worker 必须能读取一个不可变的任务配置快照，而不是依赖某台 API 节点的本地内存或环境变量。第 2、3 项应协同设计状态机和租约机制。第 4 项的结论为第 5 项提供基线。

## 2. Redis 多节点任务处理

### 目标

用 Redis 队列替代 API 进程内的单线程唤醒队列，使多个独立 worker 节点可以并行处理分析任务，同时以 PostgreSQL 作为任务状态、结果和审计记录的事实来源。

### 功能项

- [ ] 选定并封装队列实现（建议 Redis Streams 或成熟任务框架）；定义任务消息最小载荷为 `job_id`、schema 版本和投递时间，完整请求和配置仍从 PostgreSQL 读取。
- [ ] 将 API 提交流程调整为：事务写入 `analysis_jobs(status=queued)`，再可靠投递 Redis；处理数据库写入成功但投递失败的补偿与重投。
- [ ] 提供独立 worker 入口和部署参数：队列地址、consumer group/worker 名称、并发数、优雅停机超时和节点标识。
- [ ] 在 claim 时使用原子状态转换，确保同一 `job_id` 只被一个 worker 执行；取消现有全局 PostgreSQL advisory lock 的单 worker 假设，改为仅保护确有全局互斥需求的资源。
- [ ] 增加任务租约/心跳、可见性超时或 pending reclaim；worker 崩溃后由其他节点接管，并限制重复执行造成的副作用。
- [ ] 定义失败重试策略：可重试错误、最大次数、指数退避、死信队列和人工重放流程；业务校验错误不可无限重试。
- [ ] 保留进度和事件写入 PostgreSQL 的接口，并保证多节点下事件时间、状态迁移和最终结果一致。
- [ ] 提供队列积压、执行时长、重试、死信、worker 心跳和任务成功率等指标与告警。
- [ ] 更新 API 服务文档、部署示例和本地开发方式；Redis 不可用时 API 必须拒绝新任务或明确降级，不得悄悄回退到不可扩展的内存队列。

### 验收标准

- 两个 worker 同时运行时，可并行处理不同任务，同一任务最多执行一次最终业务逻辑。
- worker 进程被终止后，未完成任务在规定租约超时后可被其他 worker 接管，并保留原任务 ID 和审计事件。
- Redis、数据库和 worker 任一依赖短暂不可用时，任务不会被静默丢失；失败状态、重试次数和死信原因可查询。
- 覆盖重复投递、并发 claim、worker 中断、重试、死信和重启恢复的集成测试。

## 3. 分析任务暂停、恢复与重试

### 目标

允许用户安全地暂停排队或执行中的分析任务，在合适的执行边界恢复，并对可恢复失败进行显式、可审计的重试；该能力必须与 Redis 多节点消费及 PostgreSQL 任务状态一致。

### 功能项

- [ ] 扩展任务状态机，至少区分 `queued`、`running`、`pausing`、`paused`、`retrying`、`succeeded`、`failed` 和 `cancelled`；定义每个状态允许的迁移及冲突响应。
- [ ] 提供暂停、恢复和重试 API/CLI 操作，并要求状态迁移使用条件更新，防止多次点击或多个节点造成重复投递。
- [ ] `queued` 任务暂停后不得被 worker claim；`running` 任务收到暂停请求后，在 Agent 节点或工具调用完成等安全检查点写入 checkpoint、释放租约并转为 `paused`。
- [ ] 明确不能安全中断的区域及最大等待时间；暂停不是线程强杀，正在进行的 LLM 或供应商 HTTP 调用可完成或超时后再响应暂停。
- [ ] 恢复任务时基于已保存的 checkpoint、任务配置快照和分析截止日期继续；没有有效 checkpoint 时按清晰策略重新执行，并在事件中标记执行方式。
- [ ] 重试仅适用于明确可重试的失败类型，例如临时网络错误、超时和供应商限流；记录重试次数、原因、触发者、前次错误和退避计划，设置最大次数并支持进入死信队列。
- [ ] 定义用户手动重试与系统自动重试的幂等行为；新的执行尝试使用同一 `job_id` 和递增 attempt 编号，保持结果、事件和成本可追踪。
- [ ] worker 在 claim、续租、完成和失败前检查暂停/取消请求，避免已暂停任务被续租或写入成功结果。
- [ ] 为暂停、恢复、重试、checkpoint 恢复和重试耗尽提供审计事件、指标和告警；任务详情需展示当前状态、attempt、暂停原因和下一次重试时间。
- [ ] 更新 API 文档，说明暂停与恢复的异步语义、预期等待时间、不可恢复错误以及是否可能产生额外 LLM/数据源成本。

### 验收标准

- 暂停排队任务后，任何 worker 都不能领取它；恢复后只投递一次并可正常完成。
- 暂停执行中任务后，任务在定义的安全检查点进入 `paused`，checkpoint、进度与已发生的 token/成本记录完整保留。
- 多个 worker 和重复 API 请求下，恢复或重试不会并行启动同一 attempt；状态迁移和事件顺序可审计。
- 临时失败按配置退避并在最大次数后进入明确终态；不可重试错误不被自动循环重试。
- 覆盖 queued/running 暂停、checkpoint 恢复、并发请求、worker 崩溃、手动与自动重试、重试耗尽和成本累计的集成测试。

## 4. 数据源准确性与优先级确认

### 目标

用可重复的样本和质量指标验证当前供应商数据，并将每个工具的优先级配置为显式、可审计的调用链。

### 功能项

- [ ] 建立数据源矩阵：按工具列出当前 `tool_vendors`、类别级 `data_vendors`、实际供应商、认证要求、市场覆盖、时区、延迟、历史可用范围和费用。
- [ ] 选取覆盖美股、港股、A 股、指数、ETF、加密资产和异常/退市标的的基准样本；对历史分析固定截止日期，检查新闻和行情没有 look-ahead bias。
- [ ] 为 OHLCV、复权、财报字段、基本面、新闻时间、内部人交易、宏观数据和证券身份定义对账规则、容差和权威参考源。
- [ ] 比较各供应商完整率、字段一致性、延迟、失败率、限流率和单次成本，输出可复现的评估报告。
- [ ] 基于结果确认默认优先级：`tool_vendors` 覆盖单一工具，`data_vendors` 提供类别级有序 fallback；显式链只能调用链内供应商，不能静默加入未配置来源。
- [ ] 对每个 fallback 记录触发原因（未配置、限流、无数据、数据校验失败、上游错误）和最终来源，供任务审计、质量分析和限流决策使用。
- [ ] 为关键数据添加运行时校验，例如标的身份、交易日、过期 OHLCV、时间边界和数据完整性；校验失败必须可触发受控 fallback 或清晰失败。

### 验收标准

- 已发布工具级数据源矩阵和默认优先级，包含选择依据与不适用范围。
- 基准样本在定义的容差内通过；差异、缺失和无法验证项有可追踪记录。
- 任意一次工具调用可查到最终数据源、fallback 链与失败原因，不记录凭据。
- 通过现有供应商路由、错误处理、ticker/标的身份、新闻日期和过期 OHLCV 相关测试，并补齐新增校验的测试。

## 5. 免费数据源限流与替换评估

### 目标

将免费供应商的请求限制转化为可观测、可控制的风险，并根据质量、容量和成本数据决定保留、缓存、限速或替换。

### 功能项

- [ ] 为每个供应商记录请求数、成功率、429/配额耗尽、响应耗时、缓存命中率、fallback 次数和估算成本；指标按工具、时间窗口和 worker 维度聚合。
- [ ] 明确每个免费数据源的公开配额、技术限制和允许用途；把这些信息、最后核验日期和责任人维护为供应商台账。
- [ ] 实施供应商级限速、并发上限和指数退避；限流错误使用既有错误语义进入已配置 fallback 链，不把 429 当作无数据。
- [ ] 为适合缓存的数据定义 key、TTL、交易日/市场时区失效规则和过期标识；历史数据必须按分析截止日期隔离，避免将未来数据写入历史任务缓存。
- [ ] 对相同 `ticker`、日期和工具的并发请求进行请求合并，减少多 Agent 重复调用。
- [ ] 为限流高峰设计降级规则：使用已验证的备用供应商、返回带来源和时效说明的缓存结果，或明确终止任务；不得伪造实时数据。
- [ ] 建立替换评估模板：覆盖率、准确性、稳定性、SLA、限流、许可、成本、接入工作量、迁移风险和退出方案。
- [ ] 设置触发阈值，例如连续限流、成功率、延迟和关键字段缺失率；达到阈值时生成评估工单并复核供应商优先级。

### 验收标准

- 压测或模拟限流时，请求速率不超过设定上限，且任务按配置正确 fallback、重试或失败。
- 仪表盘能按供应商和工具识别容量压力、缓存效果和对任务成功率的影响。
- 有至少一份针对当前高风险免费供应商的替换或保留评估，结论包含数据和复核日期。
- 缓存与请求合并不破坏历史日期边界、标的隔离或数据来源审计。

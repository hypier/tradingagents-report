# 积分计费与管理员额度调整设计

## 背景与目标

`tg-web` 当前通过 `credit_accounts` 保存用户的可用、预留和已消费分析额度，通过
`credit_ledger_entries` 记录额度流水。分析提交时固定预留 1 个额度，成功后固定核销 1 个，
失败时释放。Core 已按每次任务的实际模型 token 用量计算 `analysis_jobs.cost_usd`，但该实际
美元成本尚未参与用户额度结算。

本功能将分析额度升级为积分计费：管理员设置积分兑美元比率、成本加价率、预扣缓冲率和
冷启动预估成本；系统在任务开始前预估并预扣积分，在任务成功时按实际 AI token 美元成本
多退少补。功能同时在用户管理页提供可用积分查看和管理员人工增减入口。

系统仍是投研分析产品，不涉及真实交易、账户资金或券商下单。积分是产品内部使用额度，
不是真实货币余额。

## 已确认的业务规则

- 默认积分兑美元比率为 `100 积分 / 1 美元`。
- 默认成本加价率为 `10%`，口径为成本加价，不是销售毛利率。
- 默认预扣缓冲率为 `20%`，缓冲只影响预扣，不影响最终收费。
- 没有历史样本时，默认预估成本为 `$1.00`。
- 非零实际成本产生的积分向上取整且最低为 1；零成本任务扣 0。
- 成功任务按实际成本结算；失败任务全额退回预扣积分。
- 实际费用超过预扣和剩余余额时允许产生负余额，负余额用户不能发起新任务。
- 取消“必须有有效订阅才能分析”的限制；积分余额是分析准入条件。
- 订阅只负责按周期发放积分，管理员人工增加的积分同样可以使用。
- 默认 `$20 / $50 / $100` 套餐改为发放 `2,000 / 5,000 / 10,000` 积分。

## 范围

包含：

- 管理员维护积分计费参数，并记录配置变更审计。
- 根据同类历史成功任务成本的 P90 预估本次成本。
- 无历史样本时使用管理员配置的默认预估美元成本。
- 分析开始前预扣积分，成功时按实际成本多退少补，失败时全额退回。
- 用户账单页展示预扣、释放、实际消费与成本换算信息。
- 用户管理列表展示可用积分，并允许管理员人工增加或扣减。
- 中英文界面文案、Drizzle 迁移、Web/Core API、持久化和自动化测试。
- 同步更新默认 Stripe 套餐积分元数据以及相关产品和架构文档。

不包含：

- 用户直接购买一次性积分包、提现、现金退款或积分转账。
- 按任务执行进度实时扣点或因余额不足中途终止 LangGraph。
- 独立计费 Worker、外部队列或多节点结算。
- 修改真实 LLM 供应商账单；系统只使用 Core 已计算的 `cost_usd`。
- 批量管理员额度调整或独立用户详情页。

## 架构选择

采用“TG-web 估价和预扣，Core 在任务终态事务中结算”的分工：

- TG-web 拥有产品计费配置、额度账户、历史估算和提交准入。
- Core 已拥有实际 token 成本，并负责写入 job 终态；因此它在同一 PostgreSQL 事务中结算
  对应预留，保证 job 成功/失败与积分核销不会分离。
- PostgreSQL 仍是 job、配置、钱包、预留和账本的唯一持久化来源。

不采用 Web 轮询后结算，因为 Web 重启或轮询中断可能造成漏扣；不新增独立计费 Worker，
因为当前系统是单节点串行 job 设计，引入队列、claim、租约和重试超出本次范围。

## 计费公式

设：

- `R`：积分/美元比率，默认 `100`
- `M`：成本加价率，默认 `0.10`
- `B`：预扣缓冲率，默认 `0.20`
- `E`：历史 P90 或冷启动默认预估美元成本
- `A`：Core 保存的实际美元成本

预扣积分：

```text
reserved_points = ceil(E × (1 + B) × (1 + M) × R)
```

实际积分：

```text
actual_points = 0                              当 A = 0
actual_points = max(1, ceil(A × (1 + M) × R)) 当 A > 0
```

所有金额和比率计算使用十进制定点数，不使用二进制浮点数。最终只在积分边界向上取整。
计费配置在预扣时形成快照，任务结算使用该快照；管理员之后修改配置不影响运行中或历史任务。

示例：默认配置下，冷启动预扣为
`ceil($1.00 × 1.20 × 1.10 × 100) = 132` 积分。若实际成本为 `$0.123`，最终消费
`ceil($0.123 × 1.10 × 100) = 14` 积分，并退回 118 积分。

## 历史成本预估

分析请求生成稳定的计费签名，签名包含：

- 排序后的分析师集合。
- LLM provider、deep think 模型和 quick think 模型。
- 研究辩论轮数和风险辩论轮数。

估算器从最近 100 个签名相同、状态为 `succeeded` 且 `cost_usd > 0` 的任务读取成本，按
离散 P90 取值；不存在完全匹配的历史样本时使用 `default_estimated_cost_usd`。估算只读取
任务提交时可公开覆盖的配置，不允许客户端影响数据库连接、凭据或其他运行环境边界。

新增 `POST /api/analyses/estimate`，请求体与创建分析所需的计费相关字段一致，响应包含：

```json
{
  "data": {
    "estimatedCostUsd": "1.00000000",
    "reservedPoints": 132,
    "source": "default",
    "sampleCount": 0
  },
  "requestId": "..."
}
```

前端估价只用于展示。提交分析时服务端必须重新读取当前设置和历史成本，并在数据库事务内
计算、校验和预扣，不能信任客户端传入的预估金额。

## 数据模型

### 积分计费设置

新增单行表 `credit_billing_settings`：

- `id`: 固定主键 `default`
- `points_per_usd`: 正十进制数，默认 `100`
- `markup_basis_points`: 整数基点，默认 `1000`（10%）
- `reserve_buffer_basis_points`: 整数基点，默认 `2000`（20%）
- `default_estimated_cost_usd`: 正十进制美元金额，默认 `1.00000000`
- `updated_by_clerk_user_id`
- `created_at` / `updated_at`

新增追加写审计表 `credit_billing_setting_events`，保存每次变更前后的完整配置、操作者和时间。
不复用只面向 Stripe 凭据的 `billing_config_audit_events`。

### 额度预留

扩展 `credit_reservations`：

- 现有 `units` 继续表示预扣积分。
- `estimated_cost_usd` 保存预估基础成本。
- `pricing_snapshot` 保存比率、加价率、缓冲率、预估来源、样本数和计费签名。
- `settled_units` 保存最终消费积分；失败任务为 `0`。
- `settled_cost_usd` 保存用于结算的实际成本；失败任务仍记录成本但收费为 `0`。

新增字段允许为空，以兼容部署前已经存在或正在运行的固定 1 额度预留。Core 遇到没有计费
快照的旧预留时继续按原 `units` 结算，避免升级期间改变旧任务语义。

### 账本

预扣仍写 `reserve` 流水。成功结算写一条 `consume` 流水：

- `available_delta = reserved_points - actual_points`
- `reserved_delta = -reserved_points`
- `spent_delta = actual_points`

因此预扣高于实际时自动退款，实际高于预扣时从可用余额补扣；补扣超过余额时
`available_credits` 可以变为负数。失败结算写 `release`，退回全部预扣且 `spent_delta = 0`。

结算流水 metadata 保存：实际/预估美元成本、积分比率、加价率、缓冲率、预扣积分、最终
积分、估算来源和失败释放原因。现有唯一幂等键继续保证同一任务只结算一次。

## 分析提交与结算流程

1. 前端调用估价接口显示预计预扣积分。
2. 用户提交分析，TG-web 校验当前法律文档同意状态。
3. TG-web 在事务中读取计费设置、计算历史 P90、生成配置快照并尝试预扣。
4. `available_credits < reserved_points`（包括负余额）时返回 `402 INSUFFICIENT_CREDITS`，
   不调用 Core。
5. 预扣成功后提交 Core job，并沿用 `request_id` 幂等关联预留。
6. Core 成功落库时读取 job 的 `cost_usd` 和预留快照，计算最终积分并在同一事务内更新
   reservation、钱包和账本。
7. Core 失败落库时记录实际成本用于运营分析，但将预留全额释放，用户收费为 0。
8. 若 TG-web 提交 Core 前被明确拒绝，沿用当前逻辑立即释放预留。

删除 `reserveAnalysis()` 中的有效订阅查询。Stripe webhook 按付费周期发放套餐积分到
`period_credits`（续费前清零未用套餐积分；升级补差额；`canceled`/`unpaid`/删除时清零套餐积分）。
活动奖励写入 `bonus_credits` 并跨周期保留。`past_due` 宽限期内可继续消费，不发新周期。
退款与败诉争议仅回收未使用的套餐积分。

## 管理员计费设置

管理员支付设置页新增“积分计费”标签页，包含：

- 积分/美元比率。
- 成本加价率百分比。
- 预扣缓冲率百分比。
- 冷启动默认预估成本（美元）。
- 当前冷启动预扣积分的实时预览。
- 最近更新时间和更新管理员标识。

新增管理员接口：

```http
GET /api/admin/billing/credit-settings
PUT /api/admin/billing/credit-settings
```

更新接口只接受有限范围内的正数，百分比以基点转换后持久化。更新当前配置和写入审计事件
必须处于同一事务。管理员身份只从服务端认证上下文读取。

## 管理员人工增减积分

用户管理列表增加“可用积分”和“操作”列。列表继续从 Clerk 获取用户分页，再通过一次批量
查询合并本页用户余额；没有本地钱包的用户显示为 0。

每行“调整”按钮打开 Dialog，包含增加/扣减分段控制、1 至 1,000,000 的正整数输入、
可选备注和调整后余额预览。扣减不得使可用积分低于 0；人工调整不允许制造负余额。

新增接口：

```http
POST /api/admin/users/:userId/credit-adjustments

{
  "adjustmentId": "UUID",
  "delta": 100,
  "reason": "Optional note"
}
```

目标 Clerk 用户尚未同步时，路由先调用 `AccountRepository.syncUser()` 创建本地账户和钱包。
调整只更新 `available_credits`，并写 `entry_type = adjustment` 的幂等账本行。metadata 包含
管理员 ID 和可选备注。相同 adjustment ID 重试不重复入账；标识被用于不同用户或不同金额
时返回 `409`。

## 套餐与用户账单界面

默认套餐定义由 `20 / 50 / 100` 积分改为 `2,000 / 5,000 / 10,000` 积分。默认套餐调和
逻辑更新对应 Stripe 产品/价格 metadata，使现有默认套餐订阅在后续付费周期按新积分数发放；
自定义套餐保持管理员设置的积分数。

分析提交界面在表单有效时显示“预计预扣 N 积分”，估价加载或失败不改变服务端最终校验。
用户账单页继续显示可用、预留和已消费积分，并把账本中的预估成本、实际美元成本、预扣、
退款和最终消费展示为可扫描的明细。界面不把积分描述为现金余额。

## 错误处理与幂等

- 非管理员不能读取或修改计费设置，也不能人工调整用户积分。
- 无效配置或调整输入返回 `400`。
- 预扣余额不足返回 `402`，不创建 Core job。
- 人工扣减余额不足和幂等冲突返回 `409`。
- Core 终态转换继续使用条件更新；只有首次成功转换到终态的事务执行结算。
- 缺失实际 usage 导致 `cost_usd = 0` 时成功任务扣 0，并在流水 metadata 中保留零成本，
  不猜测或补造费用。
- 估价接口失败时前端明确提示不可估价；创建接口仍独立执行服务端估价，避免使用陈旧预览。

## 测试策略

按照 TDD 顺序覆盖：

- 纯函数：十进制公式、向上取整、最低 1 积分、零成本、加价率和缓冲率边界。
- Drizzle 仓库集成：设置更新与审计、历史 P90、冷启动、配置快照、预扣、余额不足、
  纯积分准入、人工增减和幂等冲突。
- Web API：估价、创建分析时重新估价、管理员设置权限、管理员人工调整和错误映射。
- Core 存储：成功多退、成功补扣、超额负余额、零成本、失败全退、旧预留兼容和重复终态幂等。
- 前端：计费设置表单、估价预览、用户余额与调整弹窗、账本成本明细和中英文文案。
- Stripe：默认套餐积分 metadata 更新和周期发放回归。
- 全量相关验证：Web unit/integration、Core job/infrastructure 测试、类型检查、lint、生产构建、
  本机镜像重建、Drizzle 迁移和 Compose 健康/就绪检查。

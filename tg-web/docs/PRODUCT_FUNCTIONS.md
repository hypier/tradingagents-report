# TG-web 产品功能说明

## 1. 产品定位

TG-web 是面向个人投资研究用户的多市场 AI 投研产品。用户通过订阅或管理员发放获得产品积分，围绕股票行情、分析任务和研究报告完成研究工作流。

产品只提供研究分析、市场数据和报告解读，不提供真实交易下单、券商账户接入或资金管理。

## 2. 已确认的产品决策

| 主题         | 决策                                                                                                              |
| ------------ | ----------------------------------------------------------------------------------------------------------------- |
| 身份服务     | 使用 Clerk 处理注册、登录、社交登录、密码重置和会话管理。TG-web 不实现自建认证流程。                              |
| 支付服务     | 使用 Stripe 处理订阅结账、续订、取消、换套餐、账单和支付状态。                                                    |
| 角色         | 仅保留 `user` 和 `admin` 两个角色。首个注册账号由服务端自动初始化为管理员；管理员可在用户管理页调整其他账号角色。 |
| 计费         | 套餐按周期发放积分；分析按实际 AI token 美元成本、积分汇率和加价率结算。                                          |
| 应用形态     | 用户与管理员共用同一应用与账户体系；管理员可在侧栏切换「投研菜单 / 管理菜单」，两种模式分别只展示对应导航；管理接口仍由后端强制校验。 |
| 市场与语言   | 面向多语言、多市场产品设计；界面语言与报告输出语言独立配置。                                                      |
| 报告 AI 对话 | 保留为后续功能，不纳入当前实现范围。                                                                              |

## 3. 用户工作流

```mermaid
flowchart LR
    A[通过 Clerk 登录] --> B[查看订阅与分析额度]
    B --> C[搜索并确认上市标的]
    C --> D[查看最新行情]
    D --> E[提交分析任务]
    E --> F[查看任务进度]
    F --> G[阅读和管理报告]
    G --> H[查看个人用量与订阅]
```

用户在提交分析前看到预计扣减额度。系统在服务端检查并预留额度，任务按既定的成功、失败或取消规则结算。不能仅以浏览器状态或 Stripe 页面作为额度事实来源。

## 4. 用户功能

### 4.1 身份与账户

- Clerk 托管的注册、登录、退出、社交登录、密码重置和会话恢复。
- 用户个人资料：显示名称、头像、界面语言、报告语言、时区和默认市场。
- 本地用户档案保存 Clerk 用户标识、产品偏好、订阅关联和额度账本关联；不保存密码或会话凭据。

### 4.2 订阅、支付与额度

- 套餐页展示套餐价格、计费周期、每周期发放的积分、支持市场和可用功能。
- 使用 Stripe Checkout 开始订阅；使用 Stripe Customer Portal 管理续订、取消、换套餐、付款方式、发票和账单历史。取消为期末取消：本周期权益有效，下一周期不再续订。
- 积分双余额：`period_credits`（套餐账户，按周期清零重发）与 `bonus_credits`（奖励账户：注册/邀请/管理员等，跨周期保留）；可用总额为二者之和。消费时先扣套餐账户。
- Stripe Webhook：首订/续费发放套餐积分（续费前清零未用套餐积分）；升级按套餐差额补发；`canceled`/`unpaid`/删除订阅时清零套餐积分；退款与败诉争议按比例回收未用套餐积分。`past_due` 为付款宽限，保留余额但不发新周期。
- 免费层不创建 Stripe 订阅，仅奖励账户积分；分析准入只校验可用余额门槛。
- 订阅 / 账单 / 用量为独立页面；订阅页展示套餐与当前订阅，账单页展示 Stripe 发票，用量页展示可用总额、套餐账户、奖励账户、已消耗，以及当前计费周期内的额度明细（变动数字用蓝/黄区分套餐与奖励；消费说明可点进关联报告；类型用色标；隐藏历史预留/释放流水）。
- 分析提交前校验余额门槛（不做预扣）；成功或用户取消时按实际成本扣分，系统失败不扣。
- 支付回调、额度发放、清零、回收与结算必须具备幂等标识与审计记录。

### 4.3 标的、行情与多市场

- 标的搜索：按市场、交易所、股票代码和公司名称查询，并由用户确认具体上市地。
- 股票详情：公司名称、代码、交易所、币种、最新价格、涨跌、数据来源、数据时间和交易时段状态。
- 行情展示必须标注数据来源和时间。"实时"应按数据供应商的实际延迟定义，不能将最新快照表述为无延迟行情。
- 自选股：添加、移除、排序和从收藏列表进入行情或报告。
- 分析交易所白名单管理可分析交易所（表 `analysis_exchanges`；管理端展示 `shared/exchanges.json` 全量清单并勾选启用）。
- 支持美股、港股、台股、日股等市场；实际开放范围取决于可用数据源和标的搜索覆盖。

### 4.4 分析任务与报告

- 单股分析：选择标的、分析师组合、分析日期、报告语言和允许的分析配置。
- 提交前显示本次分析的额度消耗；额度不足时引导用户查看订阅和用量。
- 任务中心：展示排队、运行、成功、失败与用户取消状态，以及进度、当前步骤、事件和失败原因。运行中可请求停止：按钮进入「停止中」并锁定；事件日志写入「已请求停止」与最终「已取消」；状态徽章显示「已取消」而非「失败」。
- 报告中心：按标的、市场、状态和日期检索历史报告；支持按自选筛选与 Markdown 导出。
- 报告详情：展示结论、各分析角色的报告内容、数据截至时间、风险提示、生成状态和本次额度记录；可将标的加入自选。

### 4.5 多语言

- 界面文案、日期、数字、货币和时区按用户界面语言与区域偏好本地化。
- 报告输出语言可与界面语言不同，作为分析任务的一部分持久化保存。
- 标的代码、交易所、货币和数据源名称保持可辨识的原始金融语义，不因翻译造成歧义。

## 5. 管理员功能

管理员与普通用户使用同一应用。管理员登录后显示以下管理入口；所有对应后端接口仍必须独立校验管理员身份。

### 5.1 运营概览

- 用户数、有效订阅数、周期内分析次数、额度消耗、任务成功率和失败率。
- Stripe 订阅收入、退款或付款失败的概要信息。
- 队列积压、分析耗时、系统错误和数据源异常的健康概览。

### 5.2 用户与订阅查询

- 查询用户基本资料、订阅状态、额度余额和到期时间。
- 查看用户的额度发放、预留、扣减、释放和调整记录。
- 查看用户分析任务、报告与失败记录，用于支持和问题定位。
- 账号状态处置，例如停用或恢复；不提供管理员角色的界面授权流程。

### 5.3 套餐与计费规则

- 管理 Stripe 产品和价格对应关系、套餐启停状态和周期额度。
- 管理分析额度消耗规则，例如按市场、分析深度或分析配置定义消耗单位。
- 查看 Stripe 事件、订阅状态同步和对账异常。
- 管理员手动调整额度时必须填写原因并生成不可修改的审计记录。

### 5.4 分析任务与运行状态

- 查询任务状态、所属用户、标的、进度、运行时长、用量、成本和错误信息。
- 对符合既有状态机规则的失败任务执行受控重试。
- 监控队列、worker、任务失败率和重试情况。
- 任务、事件和成本数据必须保留用户归属与时间线，便于审计和支持。

### 5.5 模型、数据源与系统设置

- 在管理 → 模型设置下分别配置：`/admin/llm/providers`（供应商与加密 API Key）、`/admin/llm/models`（模型列表与默认快速/深度模型）。详细设计见 [`LLM_MODELS_CONFIGURATION.md`](./LLM_MODELS_CONFIGURATION.md)。
- 管理员可在新建/编辑模型时手动同步价格与参数；开放模型供用户分析时选择。
- 管理允许开放的市场、界面语言、功能开关、维护公告和风险提示文案。
- 支付私钥等仍通过部署环境变量管理；LLM API Key 在模型配置页加密保存，接口仅返回掩码。

### 5.6 审计与监控

- 记录管理员操作、Stripe 回调、订阅状态变更、额度变动和关键任务状态迁移。
- 提供可检索的系统错误、数据源错误和支付同步错误。
- 监控任务执行、额度扣减和回调处理的异常。

## 6. 后续功能：报告 AI 对话

报告 AI 对话明确不纳入当前实现范围。后续实现时应作为独立功能设计，不影响本期分析额度的规则。

- 在报告详情中围绕当前报告提问，回答应明确引用报告中的相关内容。
- 支持会话历史、推荐追问、流式回答和额度不足提示。
- 对话额度与分析额度分别定义、分别记账，避免混淆成本和套餐权益。
- 第一版建议只基于当前报告内容回答；是否扩展到实时行情或外部检索，应在后续单独确认数据来源、时效说明、成本和风险边界。

## 7. 当前实现基线

当前 TG-web 已有以下能力，可作为上述产品功能的基础：

- 分析任务提交，可选择分析师、分析日期和报告语言。
- 运行中任务的轮询进度与事件展示；独立任务中心 `/tasks` 按用户隔离展示排队/运行/成功/失败/取消。用户主动停止后 UI 保持「停止中」直至任务结束，事件日志记录停止请求与取消结果。
- 分析报告详情及按报告分区展示；支持加入自选与 Markdown 导出。
- 报告库支持按状态、标的、交易所、日期与自选筛选。
- 输入代码后获取最新市场快照，并展示数据源、截至时间与新鲜度（不再无依据标注“实时”）。
- 市场行情 `/quotes` 与标的详情 `/stocks/:providerSymbol`：logo、最新/实时报价（TradingView SSE）、趋势图，可跳转分析与加入自选。
- 自选股：添加/移除、排序，并从列表进入行情或分析。
- 最近分析报告列表与报告页面路由。
- 界面语言支持中文与英文切换；账户 `interfaceLanguage` 与界面 i18n 双向同步；界面语言与报告输出语言独立配置；支持明暗主题切换；默认市场偏好用于搜索排序。
- Clerk 托管注册、登录、退出、密码重置、会话恢复和用户菜单；前端提供 `/sign-in` 与 `/sign-up` 路由并在认证后返回原始站内地址，BFF 对分析与市场数据接口验证 Clerk 会话。具体邮箱、密码、验证码和社交登录方式由 Clerk Dashboard 配置。
- 首个注册账号在首次建立应用会话时自动写入 Clerk `publicMetadata.role=admin`；管理员拥有受 BFF 强制鉴权保护的用户列表、搜索与角色调整功能，普通用户无法调用对应接口。
- 已登录用户可在 `/billing` 查看 Stripe 循环套餐、当前订阅状态和账单，通过 Stripe Checkout 创建订阅，并通过 Customer Portal 处理续订、取消、换套餐、付款方式和账单。
- 用户可在 `/account` 通过 Clerk 管理显示名称、头像、密码、社交账号和会话，并保存界面语言、报告语言、时区和默认市场；本地不保存密码或 Clerk 会话凭据。
- Stripe Customer ID 保存在本地用户档案并镜像到 Clerk `privateMetadata`。Webhook 验签后同步本地订阅快照；`invoice.paid` 按 invoice ID 幂等发放/清零套餐积分，升级补差，退款与争议回收未用套餐积分；应用不保存银行卡数据。
- 用量页展示可用总额、套餐账户、奖励账户、已消耗与当前周期账本明细。分析提交前只校验可用积分是否**严格大于** `analysisBalanceThreshold`（不做预扣）；成功或用户取消时按实际 `cost_usd` 与创建时冻结的汇率/加价扣分（先套餐后奖励），系统失败不扣。
- 支付回调写入 `stripe_webhook_events`，额度发放、清零、回收与消费写入不可变账本并使用唯一幂等键。分析请求 UUID 同时作为 Core `request_id`，重试不会创建重复 job。
- 管理员可在 `/admin/billing` 查看 Stripe 连接与 Webhook 配置状态，创建带周期积分、支持市场和功能元数据的循环套餐并停用价格。Stripe Secret Key 与 Webhook Secret 仅通过部署环境变量（`STRIPE_SECRET_KEY` / `STRIPE_WEBHOOK_SECRET`）配置；管理端只展示连接状态与掩码提示。
- 管理员在「系统设置」与「计费 → 分析计费」维护余额门槛、每美元积分汇率与加价倍率（`system_settings.billing`）；注册/推荐奖励在「系统设置」独立配置（`system_settings.rewards`，积分数直接配置、可独立开关）。变更写入 `admin_audit_events`。
- 管理员可通过 Stripe API 幂等初始化每月 20、50、100 美元三档标准套餐，分别在有效支付周期发放 2,000、5,000、10,000 积分；初始化会升级旧版套餐 metadata，使存量订阅在后续付款周期获得新积分，重复初始化不会创建重复 Product 或 Price，配置冲突会明确报错。
- 分析列表与详情经 `analysis_jobs.clerk_user_id` 按 Clerk 用户隔离；普通用户只能查看自己的任务与报告。管理员可通过报告详情 `/reports/:id`（`GET /api/analyses/:id`）浏览任意用户报告；取消任务仍仅限所有者。
- 管理员运营概览 `/admin`：展示注册用户数、有效订阅、周期内分析量/成功率、额度消耗与队列积压，以及 Stripe 连接健康摘要。
- 管理员用户钻取 `/admin/users/:userId`：查看订阅/额度账本与近期任务，可停用或恢复账号（Clerk ban/unban），并可按幂等键手动调整额度（写入 `adjustment` 账本）。
- `/admin/users` 显示可用积分；额度调整在 `/admin/users/:userId` 完成。
- 管理员任务列表 `/admin/analyses`：跨用户按状态/标的/用户筛选；对失败任务可发起受控重试——校验所有者余额门槛并以新 `request_id` 提交替换 job（非 Core 原地重试）。
- 系统设置：默认 LLM 模型、分析扣分汇率/加价（`system_settings.billing`），以及注册/推荐积分奖励（`system_settings.rewards`）。
- 分析交易所管理 `/admin/markets`：展示 `exchanges.json` 全量清单（按 `group` 分组），勾选后点保存写入 `analysis_exchanges` 白名单；支持「已勾选」等快捷筛选。`/api/public-config` 返回已启用交易所；创建分析时校验白名单；账户默认市场由启用交易所的 `market`（catalog `country` / CRYPTO）推导。
- 管理员模型配置：`/admin/llm/providers` + `/admin/llm/models` 维护 `llm_providers` / `llm_models`、API Key 与开放状态；默认快速/深度模型在 `/admin/settings`（系统设置键 `llm`）；用户分析页从开放目录选型。
- 操作日志 `/admin/audit`：记录管理员写操作（系统设置、分析交易所、模型、用户角色、积分调整等）；错误监控复用失败任务、Stripe webhook 失败与定价来源错误。

当前尚未实现报告 AI 对话。

分析 job 仍由 Core 统一持久化。产品侧在创建 job 时写入 `clerk_user_id` 与 `credit_pricing`；Core 在 billable 终态事务内扣积分。Core 的服务级 API 本身仍不提供终端用户鉴权。产品与计费表结构由 tg-web Drizzle 迁移维护，Core 不执行 DDL。

## 8. 架构边界

- Clerk 是身份和会话的权威来源；TG-web BFF 验证 Clerk 会话，Core 使用服务凭据且不保存 Clerk 密码或会话。
- Stripe 是订阅和支付状态的权威来源；内部额度账本是分析使用权和消费记录的权威来源。
- 分析任务、进度、报告和成本继续由 Core 的既有 job 生命周期负责；TG-web 不绕过 Core 直接修改分析任务状态。
- 标的输入必须先解析为确定的交易所和代码，不以单一数据供应商的内部代码作为产品统一标识。
- 所有用户数据、报告、任务、额度和管理接口必须按用户身份或管理员身份在服务端强制隔离。

## 9. 非目标

- 真实交易下单、券商账户、资金账户或自动执行策略。
- 自建账号密码、找回密码、会话刷新或社交登录体系。
- 多层级运营角色、动态角色授权或权限配置界面。
- 将供应商凭据暴露给用户或管理员。
- 在未定义数据延迟、来源和市场覆盖前宣称提供无延迟实时行情。
